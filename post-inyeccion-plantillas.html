Inyección de Plantillas del Lado del Servidor (SSTI)

Cuando las plantillas dejan de ser seguras

SSTI ocurre si una app inserta datos del usuario en la plantilla sin validación. Eso abre la puerta a ejecutar código que ni imaginas.

Ejemplo sencillo

Plantilla: Hola {{nombre}}
Entrada: Juan → Hola Juan

Entrada maliciosa: {{7*7}} → Hola 49

Ver “49” en lugar del código significa que tienes un problema.

Cómo van las plantillas por dentro

Normalmente solo muestran datos:

template = "Hola {{usuario}}"
render(usuario="Ana") → "Hola Ana"

El fallo viene cuando el usuario puede reescribir la plantilla.

Motores con historial de vulnerabilidades

  • Jinja2 (Python): se pueden exponer objetos internos como config.
  • Twig (PHP): en versiones antiguas permitía ejecución arbitraria.
  • Handlebars: ciertos helpers inseguros pueden permitir acceso al sistema.
  • Pug / Jade: errores en nesting o lógica pueden colarse.

Tácticas típicas de ataque

  • Leak de variables internas: pedir listas de objetos, clases o entorno.
  • RCE directo: calcular resultados o leer ficheros como /etc/passwd.
  • Subida de ficheros: usar out o read para cargar shell.
  • Lista de templates: ver rutas cargables y acceder a plantillas privadas.

Maneras comunes de probarlo

  • Metes {{7*7}} en formularios, URLs o headers para ver si evalúa.
  • Intentas {{''.__class__.__mro__}} para ver clases internas.
  • Combinas payloads con {{config}}, {{os.popen}} o herramientas FP.
  • Uso de Burp Suite o Proxy para manipular valores en JSON o templates.

Otros métodos que se olvidan

  • Inyectar en plantillas de errores: algunos frameworks muestran texto de excepción con variables.
  • Reescrituras indirectas: si renderizas plantillas desde strings precargadas que el usuario puede elegir.
  • Exposición por logs: si el sistema refleja valores del input sin filtrar.

Cómo comprobar si eres vulnerable

  • Prueba primero {{7*7}} y si ves 49, hay que actuar.
  • Envía payloads tipo {{config.items()}} o {{os.environ}}.
  • Si puedes ver rutas del servidor, es señal de que hay problema.
  • Valida también que el error no muestre el stack completo.

Cómo evitarlo de verdad

  • Jamás renders directos con f-strings o `template_string` del usuario.
  • Usar archivos de plantilla controlados y nunca inputs sin procesar.
  • Implementar sandboxing (como Jinja2 SDBEnv) y limitar métodos.
  • Usar validaciones tipo re.match("^[a-zA-Z0-9 ]+$") si hay entradas reutilizables.
  • Deshabilitar funciones peligrosas, solo exponer lo mínimo.
